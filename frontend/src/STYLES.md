Frontend CSS Style Guide

Purpose
- Provide a compact, editable reference for how styles are organized in this project.

Edit workflow
- Edit `frontend/src/styles.css` (source). Do NOT edit files in `frontend/dist/` — those are generated by the build.
- Rebuild with:

```bash
cd frontend
npm run build
```

Conventions
- Top-level organization: keep `styles.css` arranged in labeled sections (see the file header).
  1) CSS Variables
  2) Base / Reset
  3) Layout & Page containers
  4) Topbar / Navigation / Brand
  5) Buttons & Controls
  6) Cards / Panels / Common surfaces
  7) Landing & Showcases
  8) Grids & Utilities
  9) Metrics
 10) Section helpers (titles, notes)
 11) Policy / Forms / Auth
 12) Tables & Tags
 13) Footer
 14) Modals / Overlays
 15) Responsive tweaks

- Naming
  - Use descriptive, dash-separated class names (e.g. `.hero-card`, `.page-header`).
  - Keep modifiers small and semantic (e.g. `.tag-success`).

- Tokens
  - All colors, spacing, and important tokens live in `:root` variables for easy overrides.

- Comments
  - Use section headers and brief comments above blocks. Keep them short and actionable.

- Component-specific styles
  - Prefer keeping shared UI tokens in `styles.css`. If a component needs a lot of styles, create a component-scoped CSS/Module and import it in the component.

- Dist files
  - `frontend/dist/assets/*.css` are generated. A formatted copy exists in `frontend/dist/assets/index-CQyJArzb.css` for debugging, but rebuilds overwrite it.

Accessibility
- Ensure focus styles remain visible (`:focus` states) when editing form controls.
- Maintain contrast for text on colored backgrounds.

Maintenance notes
- When adding new layout utilities, place them in the `8) Grids & Utilities` section.
- Keep small visual tokens (shadows, radii) in `:root` for consistency.

Contact
